# 后端开发者的 Agent 全链路学习与准备指南

作为后端开发，你已有系统、存储、并发、API 设计等基础；Agent 开发需要在此基础上补「模型侧」和「Agent 特有系统设计」。按阶段拆开，便于按自己的节奏补全。

---

## 1. 你已有的、可直接迁移的能力

| 已有能力      | 在 Agent 里的对应                             |
| ------------- | --------------------------------------------- |
| 会话/状态设计 | Session、sessionKey、多轮对话的归属与生命周期 |
| 存储与检索    | 对话历史存储、记忆持久化、向量库/FTS 检索     |
| 并发与一致性  | 同一 session 并发请求、写锁、幂等、checkpoint |
| API 与鉴权    | 工具暴露为 API、按用户/租户的 allowlist、审计 |
| 可观测        | 日志、trace、指标、告警、排查单次请求         |
| 限流与熔断    | 按用户/租户限流、模型/工具超时与降级          |
| 安全与合规    | 输入校验、PII 脱敏、审计日志、数据留存        |

**心态**：Agent 的「系统层」本质还是分布式系统 + 状态机 + 策略；先把这一层稳住，再补模型与 prompt。

---

## 2. 需要补上的核心模块（按优先级）

### 2.1 第一层：LLM API 与调用方式（必补）

**目标**：能正确调用一家主流 LLM API，理解请求/响应结构和约束。

- **必会**：
  - 请求体：`messages`（role + content）、`max_tokens`、`temperature`；可选 `tools`、`tool_choice`。
  - 响应体：`content`（文本或 `tool_calls`）、`stop_reason`（`end_turn` / `tool_use`）、`usage`（input/output tokens）。
  - 流式：chunk 的拼接、`delta` 与最终 message 的对应关系。
- **选一家先打通**：OpenAI 或 Anthropic 的 Chat Completions 文档各看一遍，用 curl 或小脚本跑通「发一条消息 → 拿到回复」和「带 tools 的一轮」。
- **实践**：写一个最小「循环」：发 user message → 若返回 `tool_calls` 则执行对应函数、把结果 append 进 messages、再请求；直到 `stop_reason === 'end_turn'`。不接任何框架，纯 HTTP 即可。

**时间**：1～2 天能跑通；理解 token 计费与限流即可，不必深究模型原理。

---

### 2.2 第二层：Prompt 与 System Prompt（必补）

**目标**：理解 system / user / assistant 角色分工，能写出「可执行、可约束」的 system prompt。

- **必会**：
  - System prompt 用来定角色、能力边界、输出格式、禁忌（不代发敏感操作、不编造等）。
  - User/Assistant 交替构成多轮；Assistant 里可能含 `tool_calls`，下一轮要补上 `tool` role 的 result。
  - 幻觉与越权多数靠 prompt 约束 + 工具设计缓解，而不是单靠模型升级。
- **学习方式**：看 1～2 篇「prompt engineering」总结（OpenAI/Anthropic 官方最佳实践）；在自家小循环里改 system prompt，观察模型是否更少乱调工具、更少跑题。
- **实践**：为「只读分析」写一版、为「可执行命令但需确认」写一版；对比行为差异。不追求完美，先建立「改 prompt → 行为变化」的直觉。

**时间**：2～3 天有感觉；长期在项目里迭代。

---

### 2.3 第三层：工具（Tools）设计与实现（必补）

**目标**：能从「后端服务/内部能力」抽象出 Agent 可用的工具：名称、描述、参数 schema、执行函数、错误与鉴权。

- **必会**：
  - 工具 = 名称 + 描述（模型靠描述决定何时调用）+ 参数 JSON Schema + 执行函数。
  - 参数设计：粒度适中、类型清晰、少用自由文本；返回尽量结构化（便于模型解析和日志）。
  - 错误：执行失败时返回什么格式给模型、是否重试、是否脱敏（不把内部错误栈直接塞给模型）。
- **与后端的衔接**：你已有的 RPC/HTTP 服务可以包装成「工具」：鉴权、限流、审计在工具层做；工具内部再调现有后端。
- **实践**：设计 2～3 个工具（例如「查用户信息」「发通知」「执行只读查询」），接入上面的小循环；观察模型在什么描述下会误用、漏用，迭代描述和参数。

**时间**：约 1 周；工具设计会长期迭代，先建立「描述 + schema + 错误处理」的闭环。

---

### 2.4 第四层：会话、上下文与记忆（核心系统能力）

**目标**：能设计「多轮对话的归属」和「上下文如何不爆窗、不丢关键信息」。

- **必会**：
  - **会话**：sessionKey/sessionId 的语义；谁和谁在对话、何时新建/复用/过期/重置（参考 `notes/session.md` 的思路）。
  - **上下文**：把「当前会话的历史消息」组进每次 LLM 请求；注意顺序（时间序）、角色、以及 tool 的 result 是否要截断或摘要。
  - **窗口与压缩**：模型有 max context；超了要 prune（删旧消息或旧 tool result）、或 compact（用 LLM 做摘要再塞回）、或把「长期记忆」单独存、按需检索注入。
- **学习方式**：读一篇「context window / context management」总结；看一个开源实现（如 OpenClaw 的 session + compaction，或 LangChain memory）的代码结构，不必全读。
- **实践**：在最小循环上接「会话存储」：按 sessionKey 存消息列表；实现「只保留最近 N 轮」或「超过 K token 时丢弃最早的一轮」；再试「先摘要再保留」的简单版 compaction。

**时间**：1～2 周建立清晰模型；压缩与记忆策略会长期调优。

---

### 2.5 第五层：编排、重试与降级（系统可靠性）

**目标**：在「模型/工具会失败、会超时」的前提下，设计可运维的流程。

- **必会**：
  - 单次 LLM 调用超时、限流、5xx 的处理：重试、退避、换模型/降级。
  - 工具调用失败：是否把错误信息给模型、格式如何、最大重试次数、何时改为「向用户道歉并结束」。
  - 长 run 的终止：用户取消、整体超时、最大步数；终止后状态一致、可审计。
- **与后端的衔接**：和你熟悉的熔断、重试、超时配置一致；只是多了一层「模型」和「工具」的边界。
- **实践**：在现有小循环里加超时、最大步数、工具失败时的固定错误消息格式；观察模型在「看到错误后」是否还会死循环调同一工具，再调策略。

**时间**：约 1 周；和业务一起迭代策略。

---

### 2.6 第六层：可观测、安全与合规（工程闭环）

**目标**：能排查问题、能控风险、能应对合规要求。

- **可观测**：一次请求的 trace（消息 → LLM → 工具1 → …）；每个工具调用的耗时、入参摘要、结果码；按 session/用户/租户的 token 与调用量。
- **安全**：输入过滤、输出过滤（若需要）；工具执行的权限与审批；PII 在日志/存储中的脱敏。
- **合规**：审计日志（谁在何时调了哪些工具）；数据留存与删除策略。

这些和传统后端一致，只是「调用链」里多了 LLM 和工具两步；用你现有的日志/trace/指标体系扩展即可。

---

## 3. 学习顺序与时间粗估

```
第 1 周：  LLM API + 最小 tool loop（2.1、2.2、2.3 的雏形）
第 2 周：  System prompt 打磨 + 2～3 个工具完整设计（2.2、2.3）
第 3～4 周：会话存储 + 上下文截断/压缩（2.4）
第 5 周：  重试、超时、降级、步数上限（2.5）
持续：     可观测、安全、合规（2.6）；以及和业务一起调 prompt/工具/策略
```

若每天只有少量时间，可拉长到 2～3 个月；重点是「先跑通最小闭环，再逐步加会话、记忆、可靠性」。

---

## 4. 推荐的学习方式（不依赖现有笔记）

- **动手优先**：用你最熟的语言（Go/Java/Python/Node）写一个「无框架」的最小 Agent 循环（HTTP 调 LLM + 内存里 2 个工具 + 控制台交互）。能跑通再考虑框架。
- **读代码**：选一个开源 Agent 实现（如本仓库的 OpenClaw，或 LangChain 的 Agent 示例），重点看：session 如何键、消息列表如何存、何时 prune/compact、工具如何注册与鉴权。不必全读，按「会话」「记忆」「工具」三条线跳读。
- **文档**：一家 LLM 的 API 文档（OpenAI/Anthropic）+ 一家「Agent 框架」的官方介绍（LangChain 或 Claude Agent SDK）足够入门；不必追很多课。
- **与现有工作结合**：若有内部系统（工单、客服、运维），想一个「Agent 能替代/辅助」的小场景，用最小闭环 + 1～2 个工具做 PoC；在真实需求里练 prompt、工具设计和错误处理。

---

## 5. 不必急于深入的部分（可后补）

- **模型原理**：Transformer、训练、微调可后补；先会用 API、会算 token、会看 `usage` 即可。
- **向量库/Embedding**：做长期记忆检索时再学；先做「最近 N 轮」或简单关键词也能跑通很多场景。
- **多 Agent/图编排**：先单 Agent 单轮/多轮跑稳，再考虑多 Agent 或 LangGraph 式编排。
- **框架深度**：先「无框架」或「只用一个框架的极小 subset」把链路打通，再决定是否深度依赖某框架。

---

## 6. 自检清单（用来判断「是否具备全链路基础」）

- [ ] 能不用框架、用 HTTP 调通一家 LLM，并实现「有 tool_calls 就执行再续传」的循环。
- [ ] 能写一个 system prompt，让模型在给定工具下「少乱用、少越权」。
- [ ] 能设计并实现至少 2 个工具（名称、描述、schema、执行、错误返回）。
- [ ] 能说清楚 sessionKey/sessionId 的语义，以及「多轮消息如何存、如何截断或压缩」。
- [ ] 能描述「工具超时/失败时」和「模型限流/超时时」的处理与重试策略。
- [ ] 能画出一条请求的 trace（消息 → LLM → 工具 → LLM → …），并知道在哪里打日志和指标。

全部打勾后，你已经具备 Agent 全链路的基础；剩下的是在具体业务里迭代 prompt、工具、记忆策略和可观测。作为后端，你的优势是「系统侧」；补上 LLM 调用与 prompt/工具设计后，可以独立负责或主导一条 Agent 产品线的系统设计与实现。
