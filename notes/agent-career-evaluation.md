# 面试者视角：如何筛选有前景的 Agent 团队与方向

Agent 方向很热，岗位名称都带「Agent」「LLM 应用」「AI 工程师」，但实际在做的事、团队深度、产品前景差异很大。从面试者角度，可以从下面几个维度判断「值不值得去」「是不是在跟风」。

---

## 1. 问题是否清晰，而非泛泛的「做 Agent」

**值得关注的信号：**

- 能说清楚**具体解决谁的问题**：哪类用户、什么场景、替代/增强的是现有哪类工作流。
- 有**可衡量的成功指标**：不是「让 Agent 更智能」，而是留存、完成率、人工介入率、错误率、成本 per task 等。
- 对**边界有认知**：知道「什么不该让 Agent 做」「什么必须人审」「什么用规则比用模型更稳」。

**需要警惕的：**

- 只说「我们做 Agent / 多模态 / 具身智能」，讲不清第一个付费场景或第一个 10x 改进点。
- 把「接上 LLM、能对话」就当产品，没有闭环（没有工具、没有记忆、没有和业务系统打通）。
- 对幻觉、延迟、成本、安全没有清晰应对思路，只说「模型会越来越好」。

**面试时可问：**  
「你们现在最想先打透的一个场景是什么？成功的话，用户/业务侧会怎么感知？」  
「如果 3 个月内只能做好一件事，会是哪一件？」

---

## 2. 技术栈是「真在做系统」还是「调 API + prompt」

**值得关注的信号：**

- 有**会话/状态/记忆**设计：多轮对话、上下文管理、压缩与检索、和业务数据打通；能讲清楚 trade-off（例如为什么用这类 memory、为什么不用另一种）。
- 有**工具与编排**：工具如何注册、鉴权、限流、错误回传；是否有编排层（线性链 vs 图 vs 自研）、何时重试、何时降级。
- 有**可观测与运维**：日志、trace、指标、告警；能复现问题、能做回归；对延迟和成本有监控。
- 有**工程边界**：安全（权限、审计、PII）、多租户、限流、合规（数据留存、删除）。

**需要警惕的：**

- 技术讨论停留在「用哪个模型」「prompt 怎么写」，对上下文爆炸、工具失败、多步漂移没有设计。
- 没有持久化状态、没有会话概念，每次请求都是无状态调用；却声称做「Agent 产品」。
- 演示很炫，但问「线上流量、错误率、人工兜底比例」答不上来或含糊。

**面试时可问：**  
「用户连续问 10 个问题，你们如何保证不丢上下文、不超窗？具体用了什么机制？」  
「工具调用失败或返回异常时，系统怎么处理？有重试/降级策略吗？」  
「线上一次请求平均经过几次 LLM 调用、多少 token？有监控吗？」

---

## 3. 团队组成：是否有人真懂「系统 + 模型」

**值得关注的信号：**

- 有**后端/基础设施**背景的人参与核心设计：会话、存储、并发、可观测、安全，而不是全交给「算法/研究员」。
- 有**产品/业务**的人一起定义场景和成功指标，而不是技术自嗨。
- 对**不确定性**有共识：知道 Agent 行为非完全确定，所以在设计上强调可观测、可干预、可降级、可审计。

**需要警惕的：**

- 团队清一色「大模型/ NLP 出身」，没人做过高可用系统、多租户、合规；却要做 toB 或高敏感场景。
- 核心产品逻辑全部绑在「模型效果」上，没有规则层、没有人工兜底、没有清晰的责任边界。
- 对「为什么不用 LangChain 一把梭」没有自己的回答（可以是「我们用了但只当组件」，也可以是「我们自建因为要满足 XX 需求」），说明对架构没有深入思考。

**面试时可问：**  
「Agent 这条线里，负责会话/记忆/工具链的是谁？和模型调优是怎么配合的？」  
「出现一次严重错误（例如误发消息、误执行命令）时，从发现到修复的流程是怎样的？」

---

## 4. 商业与节奏：有没有验证路径，而非只烧钱

**值得关注的信号：**

- 有**明确的验证路径**：先做内用、做标杆客户、做垂类，再扩张；或先做工具/平台，再做场景。
- 对**成本结构**有数：token 成本、算力、人工标注/审核占比；知道在什么规模下能算过账。
- **融资与阶段**匹配：早期看方向与团队；成长期看留存与复购、PMF 信号；不盲目堆人堆场景。

**需要警惕的：**

- 一上来就「全场景、全行业 Agent」，没有聚焦和优先级。
- 对「谁愿意为这个付钱、付多少」没有假设或验证，只说「市场很大」。
- 节奏混乱：要么长期 demo 不落地，要么一上来就冲量忽视质量和安全。

**面试时可问：**  
「目前收入/用量主要来自哪几类客户或场景？接下来 6 个月最想验证的假设是什么？」  
「单次对话/单任务的成本大概在什么量级？有在优化吗？」

---

## 5. 与你自身的匹配度

**可以顺带想清楚的：**

- **你想做系统还是做模型**：Agent 方向里，系统（会话、记忆、工具、编排、安全）和模型（prompt、微调、推理优化）是两条不同的能力线；选团队时看哪条是主战场、你能否在那条线上成长。
- **你想做 0→1 还是 1→10**：早期团队更偏探索和全栈，成熟团队更偏某一块深度；看你的阶段和偏好。
- **场景是否让你有积累**：通用聊天和垂直场景（客服、代码、运维、销售）对「领域知识 + Agent 能力」的要求不同；选一个能形成复利的赛道。

---

## 6. 小结：几句可操作的判断

| 维度     | 更可能有前景                     | 更需谨慎                        |
| -------- | -------------------------------- | ------------------------------- |
| 问题定义 | 具体场景、可衡量指标、边界清晰   | 泛泛的「做 Agent」、无闭环      |
| 技术深度 | 会话/记忆/工具/可观测/安全有设计 | 只谈模型和 prompt、无状态无运维 |
| 团队     | 系统 + 模型 + 产品都有对应的人   | 只有模型侧、无运维与合规意识    |
| 商业     | 有验证路径、对成本有数、节奏清晰 | 全场景铺、无付费验证、节奏混乱  |
| 匹配度   | 你能在系统/模型/场景上形成复利   | 岗位名是 Agent，实际只写 prompt |

**一句话：**  
有前景的 Agent 团队，会认真对待「不确定性」——用清晰的问题边界、可观测可干预的系统、合理的成本与商业验证来驾驭它；而不是只靠「模型会越来越好」和堆场景。面试时多问「具体怎么做的」「线上怎么样」「谁为结果负责」，能很快筛出是真在做系统还是在追概念。
