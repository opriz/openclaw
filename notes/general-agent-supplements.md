# 通用 Agent 补充清单：框架与难点之外还要考虑什么

在「要不要用框架、核心问题、难点」之外，做一个通用 Agent（能接工具、有记忆）时，通常还需要在下面这些维度做设计和取舍。按类归纳，便于查漏。

---

## 1. 安全与护栏

- **输入/输出过滤**：用户输入与模型输出是否需要内容过滤（违规、敏感信息）；工具返回给模型前是否要脱敏或截断。
- **越狱与滥用**：prompt 注入、越狱尝试的检测与缓解；对「代发消息、代执行命令」类工具做二次确认或权限约束。
- **工具执行边界**：哪些工具允许动生产数据、动钱、发外网；沙箱、网络隔离、文件系统白名单等。
- **PII 与日志**：对话和工具参数里是否含手机号、邮箱等；日志/追踪里如何脱敏、保留多久。
- **限流**：按用户、按 session、按 IP 的 QPS/并发限制，防止单点拖垮或滥用。

---

## 2. 多租户与隔离

- **数据隔离**：多用户/多团队时，session、记忆、对话历史是否按 tenant 严格隔离；存储与检索的 key 是否带 tenant 维度。
- **配额与计费**：按租户/用户的 token 用量、调用次数、工具调用次数；超限后的行为（拒绝、排队、降级）。
- **滥用与风控**：单租户异常流量、恶意 prompt 模式、批量注册等的发现与处置。

---

## 3. 流式、取消与体验

- **流式输出**：回复是否流式返回；工具执行期间是否先流式说明「正在执行某工具」。
- **中途取消**：用户中断后，当前 LLM 调用/工具调用是否可取消；取消后 session 状态是否一致。
- **并发请求**：同一 session 短时间内多条消息的处理策略——排队、拒绝、还是合并/覆盖；顺序与一致性如何保证。
- **超时**：单轮总超时、单次 LLM 调用超时、单次工具调用超时；超时后的提示与状态恢复。

---

## 4. 模型与路由

- **多模型/多后端**：是否支持切换模型（按会话、按请求、按能力）；路由策略（默认模型、按能力选、按成本选）。
- **降级与容错**：主模型不可用时的降级链；重试策略（退避、换模型、换 region）。
- **能力路由**：需要视觉时选带 vision 的模型；需要长上下文时选大窗口模型；在延迟与成本间权衡。

---

## 5. 工具结果的规模与截断

- **大输出**：工具返回很大（如读大文件、大表），直接塞进 context 会爆窗或浪费。
- **策略**：按 token/字符截断、先做摘要再喂给模型、或只给「摘要+指针」让模型按需再查；对「读文件」类工具可限制行数/字节。
- **结构化 vs 自然语言**：工具返回结构化（JSON）往往比大段自然语言更省 token、更易被模型利用；同时要定义错误码与错误信息规范。

---

## 6. 可观测与运维

- **指标**：延迟（p50/p99）、token 消耗、工具调用次数与失败率、session 创建/复用率、记忆检索命中率等；按 channel、agent、tenant 拆解。
- **追踪**：一次请求的完整链（消息 → LLM → 工具1 → LLM → 工具2 → …）；span 与 trace id 与日志关联。
- **告警**：错误率突增、延迟恶化、限流触发、单 session 异常长 run；与 on-call 或工单打通。
- **调试**：复现某次对话（按 sessionId + 消息 id）；导出该次完整 context（含被 prune 前的历史）便于离线分析。

---

## 7. 测试与评估

- **非确定性**：同一输入多次运行路径可能不同；单测更适合测「工具层、记忆层、会话层」的确定性逻辑，而非端到端固定输出。
- **回归与金标**：用历史对话或构造用例做回归；用关键指标（是否调对工具、是否在 N 步内完成）而非逐字对比。
- **人工评估**：抽样标注「回答质量、工具使用是否合理、是否越权」；用于调 prompt、调工具描述、调策略。
- **Canary / A/B**：新模型、新 prompt、新工具上线时小流量对比，再全量。

---

## 8. 合规与审计

- **审计日志**：谁在何时对哪个 session 发了什么、调了哪些工具、结果是否成功；保留时长与访问控制。
- **数据属地与保留**：对话与记忆存哪个 region、保留多久、用户删除账号后如何抹除。
- **合规要求**：行业或地区对 AI 交互、数据留存、人工复核的要求；是否要做内容审核、可解释性说明。

---

## 9. 扩展与部署

- **水平扩展**：gateway/worker 无状态还是带 session 亲和；session 与记忆存共享存储（DB/Redis）以便多实例共享。
- **冷启动**：首条消息要拉取 session、记忆、配置；预热、连接池、缓存策略。
- **版本与兼容**：agent 配置、prompt、工具 schema 的版本管理；向后兼容与灰度发布。

---

## 10. 用户侧体验与产品

- **发现能力**：用户如何知道能做什么（命令列表、技能列表、工具简介）；帮助文案与示例。
- **反馈闭环**：点赞/点踩、纠正、 regenerate；是否回写记忆、是否用于微调或 prompt 优化。
- **多语言**：回复语言与用户偏好一致；工具名/描述是否本地化、是否影响模型选择工具的稳定性。
- **首次使用**：引导、默认模型与默认工具、隐私与条款同意；避免第一次就用高危工具。

---

## 11. 恢复与一致性

- **中途失败**：LLM 调用或工具执行失败、超时、进程崩溃；是否支持从上一稳定状态重试、是否做部分写入与回滚。
- **幂等**：同一消息重复投递（如 MQ at-least-once）时，是否去重；工具若写库是否做幂等或补偿。
- **检查点**：长 run 是否在关键步骤落 checkpoint，崩溃后从 checkpoint 恢复而非整轮重跑（与 LangGraph checkpoint 类似思路）。

---

## 12. 小结：可当检查表用

| 类别       | 典型问题                         |
| ---------- | -------------------------------- |
| 安全与护栏 | 过滤、越狱、工具边界、PII、限流  |
| 多租户     | 隔离、配额、计费、风控           |
| 体验       | 流式、取消、并发请求、超时       |
| 模型       | 多模型、降级、能力路由           |
| 工具结果   | 大输出截断、摘要、结构化         |
| 可观测     | 指标、追踪、告警、调试复现       |
| 测试评估   | 非确定性、回归、人工评估、Canary |
| 合规审计   | 审计日志、留存、删除、合规要求   |
| 扩展部署   | 水平扩展、冷启动、版本兼容       |
| 产品体验   | 能力发现、反馈、多语言、首次使用 |
| 恢复一致性 | 中途失败、幂等、检查点           |

做通用 Agent 时，在「循环 + 工具 + 记忆」之外，按上面清单过一遍，能减少后期在安全、合规、运维和体验上的返工。
