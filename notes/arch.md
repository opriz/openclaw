# Agent 架构笔记

## 1. Agent 设计的核心概念

### 1.1 感知-决策-行动循环

```
┌─────────────────────────────────────────┐
│           Agent Core Loop               │
│                                         │
│  ┌─────────┐    ┌─────────┐    ┌─────┐ │
│  │ Percept │───▶│  Think  │───▶│ Act │ │
│  │  感知    │    │  决策    │    │ 行动│ │
│  └─────────┘    └─────────┘    └─────┘ │
│       ▲                            │    │
│       └────────────────────────────┘    │
│           反馈 / 状态更新                │
└─────────────────────────────────────────┘
```

### 1.2 核心组件

| 组件          | 作用                                   |
| ------------- | -------------------------------------- |
| Tool Use      | Agent 与外部世界交互的能力边界         |
| Memory        | 短期(对话历史) + 长期(向量存储/知识库) |
| Planning      | 将复杂任务分解为可执行步骤             |
| Reflection    | 自我评估和纠错能力                     |
| System Prompt | 定义 Agent 的角色、目标和约束          |

### 1.3 设计本质

**Agent = LLM + Tools + Memory + Feedback Loop**

- **LLM**：提供推理和理解能力
- **Tools**：扩展能力边界(如搜索、代码执行、API调用)
- **Memory**：维持上下文和经验积累
- **Feedback**：从执行结果中学习，形成闭环

### 1.4 关键设计决策

1. 自主性程度：完全自动 vs 需要人类确认
2. 工具粒度：细粒度(原子操作) vs 粗粒度(复杂功能)
3. 记忆策略：无状态 vs 有状态 vs 向量检索
4. 多 Agent 协作：单 Agent vs 多 Agent 分工

## 2. 从控制论和信息论角度看 OpenClaw 的 Memory 设计

### 2.1 控制论视角：反馈系统与稳定性

控制论核心是通过反馈实现目标导向的自适应系统。Memory 在这里是系统的状态变量存储。

#### 2.1.1 控制回路结构

```
     ┌─────────────────────────────────────────────────────┐
     │                   REFERENCE (目标)                   │
     │              "保持对话连贯性" + "控制Token成本"        │
     └───────────────────────────┬─────────────────────────┘
                                 │
     ┌───────────────────────────▼─────────────────────────┐
     │                  COMPARATOR (误差检测)                │
     │      contextWindow - usedTokens - reserveTokensFloor  │
     └───────────────────────────┬─────────────────────────┘
                                 │
                 ┌───────────────┴───────────────┐
                 │                               │
     ┌───────────▼───────────┐      ┌───────────▼────────────┐
     │   CONTROLLER (控制器)   │      │  DISTURBANCE (扰动)    │
     │  - Prune Context       │      │  - 新消息涌入           │
     │  - Compact History     │      │  - 并发写入             │
     │  - Memory Flush        │      │  - 文件损坏             │
     └───────────┬───────────┘      └────────────────────────┘
                 │
     ┌───────────▼───────────────────────────────────────────┐
     │              PROCESS (被控对象: 对话上下文)            │
     │  SessionStore + Transcript (sessions.json + .jsonl)   │
     └───────────┬───────────────────────────────────────────┘
                 │
     ┌───────────▼───────────────────────────────────────────┐
     │              SENSOR (传感器: 测量)                     │
     │  - Token计数器 (inputTokens, outputTokens, total)      │
     │  - updatedAt时间戳                                     │
     │  - contextPruning.cacheTTL                            │
     └───────────┬───────────────────────────────────────────┘
                 │
                 └───────────────┐
                                 │
     ┌───────────────────────────▼─────────────────────────┐
     │               FEEDBACK (反馈信号)                     │
     │        usedTokens → Controller → 修正动作             │
     └─────────────────────────────────────────────────────┘
```

#### 2.1.2 关键控制机制

| 控制机制         | 控制论概念   | 实现方式                      |
| ---------------- | ------------ | ----------------------------- |
| History Limiting | 前馈控制     | 预先限制历史轮次,防止问题发生 |
| Context Pruning  | 比例控制     | 根据TTL动态修剪 tool results  |
| Compaction       | 阈值触发控制 | 超过 softThreshold 时触发压缩 |
| Memory Flush     | 补偿控制     | 在压缩前主动持久化重要信息    |
| Write Lock       | 互斥锁       | 防止并发写入导致状态不一致    |
| File Repair      | 纠错控制     | 检测并修复损坏的状态          |

#### 2.1.3 稳定性分析

**潜在不稳定性来源:**

1. 时间延迟 - 从状态变化到控制动作的延迟
2. 并发冲突 - 多进程同时写入同一 session
3. 估计误差 - Token 估算不准确导致过早/延迟压缩

**设计保证:**

- `.lock` 文件确保原子性
- TTL 缓存减少陈旧读取
- 1.2x 安全余量应对 Token 估计误差

### 2.2 信息论视角：熵、压缩与信道容量

信息论关注信息的度量、存储和传输效率。

#### 2.2.1 信息流动与损失

```
     ┌─────────────────────────────────────────────────────────────────┐
     │                    INFORMATION FLOW                             │
     ├─────────────────────────────────────────────────────────────────┤
     │   User Message                                                   │
     │        │                                                         │
     │        ▼                                                         │
     │   ┌─────────────────────────────────────────────────┐           │
     │   │  Channel 1: Full Transcript (.jsonl)             │           │
     │   │  - Lossless (append-only)                        │           │
     │   │  - Entropy: 高 (完整保留所有信息)                 │           │
     │   └─────────────────────────────────────────────────┘           │
     │        │                                                         │
     │        ▼                                                         │
     │   ┌─────────────────────────────────────────────────┐           │
     │   │  Channel 2: Session Metadata (sessions.json)     │           │
     │   │  - Token counts, timestamps, metadata            │           │
     │   │  - Entropy: 低 (仅摘要)                          │           │
     │   └─────────────────────────────────────────────────┘           │
     │        │                                                         │
     │        ▼                                                         │
     │   ┌─────────────────────────────────────────────────┐           │
     │   │  Channel 3: LLM Context (传递给模型)             │           │
     │   │  - Lossy (pruning + compaction)                 │           │
     │   │  - Entropy: 动态调节 (自适应)                    │           │
     │   └─────────────────────────────────────────────────┘           │
     │        │                                                         │
     │        ▼                                                         │
     │   ┌─────────────────────────────────────────────────┐           │
     │   │  Channel 4: Compaction Summary                  │           │
     │   │  - 高度压缩的语义摘要                             │           │
     │   │  - Entropy: 极低但信息密度高                     │           │
     │   └─────────────────────────────────────────────────┘           │
     │        │                                                         │
     │        ▼                                                         │
     │   ┌─────────────────────────────────────────────────┐           │
     │   │  Channel 5: Memory Files (MEMORY.md)            │           │
     │   │  - 语义级别持久化 (人工筛选)                      │           │
     │   │  - Entropy: 极低 (仅关键事实)                    │           │
     │   └─────────────────────────────────────────────────┘           │
     └─────────────────────────────────────────────────────────────────┘
```

#### 2.2.2 信息压缩层次

```
信息密度 (bits/semantic unit)
───────────────────────────────────────
高 │ .jsonl 完整记录
   │
   │ LLM Context (修剪后)
   │
   │ Compaction Summary (LLM生成摘要)
   │
低 │ Memory Files (人工筛选)
   │
   └─────────────────────────────────
```

#### 2.2.3 互信息

互信息 I(X;Y) 衡量两个变量之间的共享信息:

| 变量对                        | 互信息 | 说明                          |
| ----------------------------- | ------ | ----------------------------- |
| 原始对话 ↔ Compaction Summary | 中     | 保留语义,丢失细节             |
| 原始对话 ↔ Pruned Context     | 高     | tool results 被修剪但语义保留 |
| Pruned Context ↔ LLM Response | 高     | 这是模型实际看到的            |
| Memory Files ↔ 当前对话       | 低     | 显式查询时才加载              |

### 2.3 OpenClaw Memory 设计的信息论特征

#### 2.3.1 渐进式有损压缩

```
阶段0: 原始消息 (全部保留)
  ↓
阶段1: Context Pruning (移除 tool results, TTL=5m)
  ├─ 信息损失: 低 (tool results 通常可重新生成)
  └─ 压缩比: ~30-50%
  ↓
阶段2: History Limiting (保留最近 N 轮)
  ├─ 信息损失: 中 (早期对话可能丢失上下文)
  └─ 压缩比: 取决于 N
  ↓
阶段3: Compaction (LLM 摘要)
  ├─ 信息损失: 高 (不可逆,细节丢失)
  └─ 压缩比: ~10-20% (保留语义骨架)
  ↓
阶段4: Memory Flush (人工/LLM 筛选持久化)
  ├─ 信息损失: 极高 (仅最关键事实)
  └─ 压缩比: ~1-5% (高密度知识)
```

#### 2.3.2 信噪比 (SNR) 管理

- **信号**: 对话中的语义信息
- **噪声**: 重复内容、过期的 tool results、冗余细节

| 操作            | 信号保留                 | 噪声抑制               | SNR 变化 |
| --------------- | ------------------------ | ---------------------- | -------- |
| Context Pruning | 高 (保护 user/assistant) | 高 (清除 tool results) | 提升     |
| Compaction      | 中 (语义摘要)            | 高 (细节丢失)          | 提升     |
| Memory Flush    | 低 (仅关键)              | 极高                   | 极高     |

### 2.4 设计优缺点分析

#### 2.4.1 优点

| 特性              | 控制论视角          | 信息论视角          |
| ----------------- | ------------------- | ------------------- |
| 双存储架构        | 状态分离,快慢分离   | 无损存储 + 有损传输 |
| Append-only JSONL | 状态可回溯,纠错友好 | 不破坏历史熵        |
| Tree 结构消息     | 支持分支探索        | 保留决策路径信息    |
| Token 估计自适应  | 反馈控制,误差补偿   | 动态调整信息预算    |
| Memory Flush      | 前馈保护,主动补偿   | 关键信息提前保存    |
| Write Lock        | 并发控制,状态一致性 | 防止信道冲突        |
| FTS5 + Embeddings | 快速检索(减少延迟)  | 语义相似度检索      |

#### 2.4.2 缺点/权衡

| 问题              | 控制论视角                 | 信息论视角       |
| ----------------- | -------------------------- | ---------------- |
| Compaction 不可逆 | 状态崩溃后无法完全恢复     | 信息熵永久损失   |
| Token 估计误差    | 控制精度受限于估计质量     | 可能浪费信道容量 |
| Memory 显式查询   | 需要主动控制才加载         | 互信息利用不足   |
| JSONL 线性扫描    | 检索延迟高 O(n)            | 随机访问效率低   |
| 单点故障          | sessions.json 损坏影响路由 | 元数据丢失       |
| 无版本控制        | 状态变更无回滚             | 信息变更历史丢失 |

#### 2.4.3 改进方向 (从理论角度)

**控制论改进:**

1. 引入 PID 控制而非简单阈值触发
2. 添加状态监控指标(如信息损失率)
3. 实现自适应调节(根据对话特性动态调整参数)

**信息论改进:**

1. 增量编码(delta encoding)减少存储
2. 语义哈希(semantic hashing)提高检索效率
3. 信息瓶颈(Information Bottleneck)方法优化压缩

### 2.5 核心设计哲学总结

```
┌─────────────────────────────────────────────────────────────────┐
│                     DESIGN PHILOSOPHY                            │
├─────────────────────────────────────────────────────────────────┤
│  控制论: "维持对话连贯性" + "控制资源成本"                       │
│  通过多级反馈回路,在上下文窗口约束下实现目标导向的对话管理       │
│                                                                  │
│  信息论: "渐进式有损压缩" + "语义保留优先"                       │
│  从无损记录到高度压缩的知识,分层管理信息熵,在信道容量约束下       │
│  最大化互信息                                                     │
│                                                                  │
│  两者结合: 一个自适应的、资源受限的、长期运行的对话代理状态管理系统 │
└─────────────────────────────────────────────────────────────────┘
```

这种设计在工程实用性和理论优雅性之间达到了很好的平衡,非常适合长期运行的 AI 对话系统。
