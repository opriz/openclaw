# 通用 Agent 选型与难点：类似 OpenClaw 的路线

目标：实现一个**通用 Agent**，能接入工具、有记忆管理，类似 OpenClaw。本文回答：要不要用框架、用什么框架、核心要解决什么问题、难点在哪。

---

## 1. 应不应该使用框架？

### 结论：看阶段和约束

| 阶段/目标                            | 建议                                                                     |
| ------------------------------------ | ------------------------------------------------------------------------ |
| **快速验证 / 单场景 demo**           | 用框架（LangChain 或 Claude Agent SDK）快速搭出「LLM + 工具 + 简单记忆」 |
| **多通道、长期运行、强可控**         | 框架只能当**组件**用，核心编排和记忆往往要自建（OpenClaw 即如此）        |
| **必须多模型、多后端**               | LangChain 的模型抽象有用；若只绑 Claude，Claude Agent SDK 也可考虑       |
| **强产品化（权限、审计、会话策略）** | 框架一般不覆盖，需在框架之上或之外自建                                   |

**OpenClaw 的实际情况**：没有用 LangChain/LangGraph 做主循环，而是自建 Agent 循环 + 自建 Session/Memory/Compaction，工具用统一接口（`AnyAgentTool`）注册。框架更多是「可选的集成点」（例如某条链路用 LangChain 的 RAG），而不是整条 Agent 生命周期的底座。

---

## 2. 若用框架，用什么？

（与 `notes/claude-sdk-langchain-langgraph.md` 对齐，这里只做选型摘要。）

| 需求                                                   | 更合适的选择                                                                                 |
| ------------------------------------------------------ | -------------------------------------------------------------------------------------------- |
| **通用 Agent + 工具 + 记忆，且接受绑 Claude**          | **Claude Agent SDK**：内置工具（Read/Edit/Bash/…）、Session、Hooks、MCP、Subagents，少造轮子 |
| **要接多种 LLM、做 RAG/简单 pipeline**                 | **LangChain**：组件全，链式流程够用                                                          |
| **复杂工作流、多 Agent、人工审核、断点续跑**           | **LangGraph**：图编排、Checkpoint、`interrupt_before`                                        |
| **像 OpenClaw 那样的「多通道 + 会话策略 + 记忆分层」** | 框架做「单次对话/单图」可以，**会话路由、记忆策略、compaction、回收**通常要自己实现          |

若目标是「通用 Agent，能接工具 + 记忆」，且可以绑 Claude：

- **优先试 Claude Agent SDK**，看其 Session、Permissions、工具集是否够用。
- 若需要**多模型**或**复杂 DAG 编排**，用 **LangChain + LangGraph** 做执行层，**会话与记忆仍建议自建**，和 OpenClaw 一样。

## 3. 核心要解决什么问题？

通用 Agent（类似 OpenClaw）在能力上要覆盖这几块，和用不用框架无关：

### 3.1 感知–决策–行动闭环

- **感知**：用户输入 + 当前会话历史 + 记忆检索结果 + 工具返回。
- **决策**：LLM 决定下一句回复或下一个 tool call。
- **行动**：执行工具 / 发消息 / 写记忆。

即：**Agent = LLM + Tools + Memory + Feedback Loop**（见 `notes/arch.md`）。

### 3.2 工具接入

- **统一接口**：名称、描述、参数 schema、执行函数，并能挂权限/审计（OpenClaw 的 `AnyAgentTool` + `createOpenClawTools` + 策略管线）。
- **可扩展**：插件/扩展能注册新工具，与内置工具一起参与策略过滤（allowlist/denylist、按 channel/agent 等）。
- **安全与可控**：敏感工具需权限控制、可选的审批、以及错误不直接暴露给模型时的降级策略。

### 3.3 记忆管理（核心中的核心）

- **短期**：当前对话的 transcript（消息列表），要能剪枝、压缩，避免爆 context window。
- **长期**：跨会话、可检索的记忆（如向量库、FTS、MEMORY 文件），供「回忆」再注入上下文。
- **策略**：何时写入长期记忆、何时做 compaction、何时过期/归档/回收（见 `notes/arch.md` 的控制论/信息论视角和 `notes/session.md` 的 session 生命周期）。

记忆管理不是「用一个 VectorStore」就完事，而是**分层 + 策略 + 与上下文窗口的反馈控制**（prune → compact → flush），这块框架通常只给基础能力，策略要自己设计。

### 3.4 会话与路由

- **会话标识**：谁在和谁对话（例如 sessionKey = agent + channel + thread/user），对应到一条会话和一份 transcript。
- **生命周期**：新建、复用、过期、重置（daily/idle/manual），以及归档与回收（见 `notes/session.md`）。
- **多通道**：同一逻辑 Agent 接入 Telegram、Slack、HTTP 等，路由到正确 sessionKey，并可能按 channel 做策略（如工具 allowlist）。

### 3.5 可观测与可控

- **可观测**：日志、trace、会话/运行状态查询（便于排查「为什么这次走了这条路」）。
- **可控**：终止运行、权限/审批、速率限制、审计（谁在何时调了哪些工具）。

## 4. 难点是什么？

（与 `notes/agent-frameworks.md` 的「构建 Agent 的真正难点」一致，这里按通用 Agent 视角再归纳。）

### 4.1 上下文窗口与记忆策略

- 历史越长，token 越多，最终必然触及上限。
- **难点**：在「保留语义」和「控制长度」之间做权衡——prune 哪些、compact 成什么、何时 flush 到长期记忆；参数（TTL、轮数、阈值）需要调优，且和模型、使用场景强相关。
- OpenClaw 的做法：多级反馈（pruning → compaction → memory flush），见 `notes/arch.md`。

### 4.2 工具设计

- 粒度太粗，模型不知道何时调；太细，选择困难、调用链冗长。
- **难点**：抽象边界、参数设计、返回结构（结构化 vs 自然语言）、错误如何反馈给模型（或截断），都要在实际使用里迭代。
- 另外还有：权限边界、与 session/channel 的绑定、和记忆/审计的衔接。

### 4.3 错误处理与恢复

- 工具不存在、参数错误、超时、返回异常等。
- **难点**：错误信息是否原样回给 LLM、格式如何、重试几次、何时降级为「向用户道歉并结束」；这些会直接影响稳定性和体验（OpenClaw 有 32～160 次等策略，是踩坑后的结果）。

### 4.4 不确定性与可复现性

- 同一输入，模型可能走不同路径；难以做确定性单测，也难以保证 SLA。
- **难点**：通过 prompt、工具设计、记忆注入来「约束」行为；通过日志和 trace 做事后分析，而不是追求完全确定性。

### 4.5 长任务与目标漂移

- 步骤一多，模型容易忘掉最初目标或重复已完成步骤。
- **难点**：system prompt 里强调目标与约束、在关键节点注入「当前目标/已完成项」、以及用子 Agent 或子图把长任务拆成有边界的小任务（见 `notes/task-decomposition.md`）。

### 4.6 成本与延迟

- 每步都调 LLM，延迟和成本会叠加。
- **难点**：在「少步数/小模型」与「一次做对、少重试」之间权衡；并行工具调用要考虑模型能否正确理解并行结果；压缩历史能省 token 但可能丢关键信息。

### 4.7 会话与记忆的「产品语义」

- 何时「忘记」、何时「记得」、用户如何触发重置或查看记忆，都影响体验。
- **难点**：和业务强相关——daily reset、idle 超时、显式 `/new`、记忆条目的可见/可删等，需要产品定规则，再在 session 与 memory 层实现（见 `notes/session.md`）。

## 5. 总结表

| 问题                 | 简短回答                                                                                                    |
| -------------------- | ----------------------------------------------------------------------------------------------------------- |
| **要不要用框架？**   | 原型/单场景可用框架；要做成「多通道 + 强记忆 + 强策略」的通用 Agent，核心编排和记忆建议自建，框架当组件用。 |
| **用什么框架？**     | 绑 Claude 且求快 → Claude Agent SDK；多模型/复杂编排 → LangChain + LangGraph；会话与记忆仍自建。            |
| **核心要解决什么？** | 闭环（LLM+Tools+Memory）、工具接入与策略、**记忆分层与策略**、会话与路由、可观测与可控。                    |
| **难点是什么？**     | 上下文/记忆策略、工具抽象与错误处理、不确定性、长任务漂移、成本与延迟、会话/记忆的产品语义。                |

一句话：**通用 Agent 的难点不在「写出一个 loop」或「接上某个框架」，而在在不确定性下把上下文、记忆、工具、会话和策略做成可靠、可运维的系统。** 框架能加速一部分，但记忆与会话的设计与实现往往需要自己深度投入。
